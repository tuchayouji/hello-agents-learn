# 第六章 框架开发实践

> **一句话总结**：四大框架解决不同问题——AutoGen 对话协作，AgentScope 工程化平台，CAMEL 角色扮演，LangGraph 图结构工作流

---

## ⭐ 为什么需要智能体框架

| 痛点 | 框架解决方案 |
|------|--------------|
| 代码复用差 | 通用基类和组件 |
| 模块耦合紧 | 模型层/工具层/记忆层解耦 |
| 状态管理复杂 | 内置状态管理机制 |
| 调试困难 | 事件回调、可观测性 |

---

## ⭐ 四大框架对比

| 框架 | 核心思想 | 适用场景 | 学习曲线 |
|------|----------|----------|----------|
| **AutoGen** | 对话驱动协作 | 流程化任务、软件开发 | ⭐⭐ |
| **AgentScope** | 消息驱动平台 | 大规模、高可靠性系统 | ⭐⭐⭐ |
| **CAMEL** | 角色扮演协作 | 双智能体深度协作 | ⭐ |
| **LangGraph** | 图结构工作流 | 复杂循环、反思迭代 | ⭐⭐ |

## ⭐ AutoGen

### 核心机制
- **对话驱动**：将任务映射为智能体间的自动化对话
- **角色分工**：AssistantAgent（思考）+ UserProxyAgent（执行）
- **团队协作**：RoundRobinGroupChat 轮询群聊

### 关键组件
```
┌─────────────────────────────────────────┐
│            AutoGen 架构                 │
├─────────────┬───────────────────────────┤
│ agentchat   │  高级对话接口             │
├─────────────┼───────────────────────────┤
│ core        │  底层基础（LLM、消息）    │
└─────────────┴───────────────────────────┘
```

### 实践案例
**软件开发团队**：
- ProductManager：需求分析
- Engineer：代码实现
- CodeReviewer：代码审查
- UserProxy：测试验收

### 优势与局限

| ✅ 优势 | ❌ 局限 |
|---------|---------|
| 自然的对话协作 | 对话不确定性 |
| 角色专业化分工 | 调试困难 |
| 人类在环设计 | 依赖LLM能力 |

## ⭐ AgentScope

### 核心机制
- **消息驱动**：所有交互抽象为消息发送/接收
- **工程化优先**：内置分布式部署、容错恢复
- **异步架构**：原生支持并发

### 关键组件
```
┌─────────────────────────────────────────┐
│           AgentScope 分层架构           │
├─────────────────────────────────────────┤
│  开发与部署层（Runtime、Studio）        │
├─────────────────────────────────────────┤
│  多智能体协作层（MsgHub、Pipeline）     │
├─────────────────────────────────────────┤
│  智能体基础设施层（ReAct、工具调用）    │
├─────────────────────────────────────────┤
│  基础组件层（Message、Memory、Model）   │
└─────────────────────────────────────────┘
```

### 消息标准结构
```python
Msg(
    name="Alice",           # 发送者
    content="Hello, Bob!",  # 内容
    role="user",           # 角色
    metadata={}            # 元数据
)
```

### 实践案例
**三国狼人杀**：
- 消息驱动的游戏流程
- 结构化输出约束游戏规则
- 并发处理与容错机制

### 优势与局限

| ✅ 优势 | ❌ 局限 |
|---------|---------|
| 高并发、高可靠性 | 学习曲线陡峭 |
| 原生分布式支持 | 简单场景过度工程化 |
| 强大的容错机制 | 生态待完善 |

## ⭐ CAMEL

### 核心机制
- **角色扮演**：两个智能体扮演互补角色
- **引导性提示**：初始提示注入行动纲领
- **自主协作**：最少人类干预

### 角色设定
```
AI User（需求方）：提出需求、下达指令
    ↕ 对话协作
AI Assistant（执行方）：执行操作、提供方案
```

### 引导性提示要素
1. 明确自身角色
2. 告知协作者角色
3. 定义共同目标
4. 设定行为约束和沟通协议

### 实践案例
**AI科普电子书**：
- 心理学家：提供专业知识
- 作家：转化为通俗文字
- 四阶段协作：框架搭建→内容生成→迭代优化→总结升华

### 优势与局限

| ✅ 优势 | ❌ 局限 |
|---------|---------|
| 轻架构、重提示 | 对提示工程依赖高 |
| 自然涌现协作行为 | 协作规模受限 |
| 多模态、多框架联动 | 任务适用性有边界 |

## ⭐ LangGraph

### 核心机制
- **图结构**：节点（计算步骤）+ 边（跳转逻辑）
- **状态机**：全局状态在节点间传递
- **条件边**：动态路由实现循环和分支

### 三要素
```
State（状态）：TypedDict，共享数据结构
    ↓
Node（节点）：Python函数，读取状态→处理→返回更新
    ↓
Edge（边）：常规边（固定跳转）/ 条件边（动态路由）
```

### 状态定义示例
```python
class AgentState(TypedDict):
    messages: List[str]      # 对话历史
    current_task: str        # 当前任务
    final_answer: str        # 最终答案
```

### 条件边示例
```python
def should_continue(state) -> str:
    if len(state["messages"]) < 3:
        return "continue"  # 跳回某节点
    else:
        return "end"       # 结束流程
```

### 实践案例
**三步问答助手**：
1. 理解（Understand）：分析用户意图
2. 搜索（Search）：搜索相关信息
3. 回答（Answer）：生成最终答案

### 优势与局限

| ✅ 优势 | ❌ 局限 |
|---------|---------|
| 天然支持循环和分支 | 需要理解图结构概念 |
| 直观的可视化工作流 | 调试链路较长 |
| 灵活的条件路由 | 相对较新，文档待完善 |

## ⭐ 框架选型指南

| 场景 | 推荐框架 | 原因 |
|------|----------|------|
| 流程化任务（如软件开发） | **AutoGen** | 对话驱动、角色分工 |
| 大规模生产系统 | **AgentScope** | 高并发、高可靠性 |
| 双智能体深度协作 | **CAMEL** | 轻量、角色扮演自然 |
| 复杂循环工作流 | **LangGraph** | 图结构、条件路由 |
| 快速原型验证 | **CAMEL** | 学习曲线低 |
| 需要反思迭代 | **LangGraph** | 天然支持循环 |

## 📝 核心代码模式

### AutoGen 模式
```python
# 定义智能体
agent = AssistantAgent(name="xxx", system_message="...")

# 创建团队
team = RoundRobinGroupChat(
    participants=[agent1, agent2, agent3],
    termination_condition=TextMentionTermination("TERMINATE"),
    max_turns=20
)

# 运行
result = await team.run_stream(task="...")
```

### LangGraph 模式
```python
# 定义状态
class State(TypedDict):
    messages: list

# 创建图
workflow = StateGraph(State)

# 添加节点
workflow.add_node("planner", planner_func)
workflow.add_node("executor", executor_func)

# 添加边
workflow.add_edge("planner", "executor")
workflow.add_conditional_edges("executor", should_continue, {...})

# 编译运行
app = workflow.compile()
result = app.invoke(inputs)
```
